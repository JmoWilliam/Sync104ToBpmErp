### ⚠️ 2.0 通用處理要求

#### 2.0.1 UPSERT 邏輯
每筆資料在寫入前 **先查詢是否存在（WHERE PK）**：
- **存在** → 執行 UPDATE
- **不存在** → 執行 INSERT

#### 2.0.2 OID 產生函數（BPM 專用）
BPM 所有 Table 的 `OID` 為 `nchar(32)` Primary Key：
```csharp
string GenerateOID()
{
    while (true)
    {
        // 產生 32 位數英數字亂碼 (A-Z, a-z, 0-9)
        const string chars = "ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789";
        var oid = new char[32];
        for (int i = 0; i < 32; i++)
            oid[i] = chars[Random.Shared.Next(chars.Length)];
        var oidStr = new string(oid);

        // 檢查系統中是否已存在
        // 需查詢所有相關 Table (Organization, OrganizationUnit, OrganizationUnitLevel, Employee, Users)
        if (!OIDExistsInDatabase(oidStr))
            return oidStr;
    }
}
```

> **查詢範圍**: Organization, OrganizationUnit, OrganizationUnitLevel, Employee, Users

#### 2.0.3 104 API 查無資料處理
調用 104 API 時，若回傳空陣列或找不到資料：
- 記錄 Log：`"[HR API] 無 {資料類型} 資料 (CO_ID={coId})"`
- **跳過**該筆資料同步，不中斷流程

## 3️⃣ 同步順序（重要！）

由於 BPM Table 之間有 FK 依賴，**同步順序必須嚴格遵守**：

```
Step 1: /api/os/company
   ├── BPM Organization.insert() ← 公司（供 OrganizationUnit 參考）
   └── (ERP 無對應公司表，或者寫入 geu_file)

Step 2.2: /api/os/dept_level (部門層級)
   ├── BPM OrganizationUnitLevel.insert() ← 層級（供 OrganizationUnit.levelOID 參考）
   └── ERP abd_file.insert()

Step 2.1: /api/os/dept (部門)
   ├── BPM OrganizationUnit.insert() ← 部門（需先有 Organization + OrganizationUnitLevel）
   └── ERP gem_file.insert()

Step 2.3: /api/ed/emp (員工)
   ├── BPM Users.insert() ← 使用者（先寫，供 Employee.userOID 參考）
   ├── BPM Employee.insert() ← 員工（後寫，需有 Users.OID + OrganizationUnit.OID）
   └── ERP gen_file.insert()
```
